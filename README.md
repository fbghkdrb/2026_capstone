# 지정학적 위기 시 암호화폐 안전자산 검증

안전자산 정의 : Baur & Lucey(2010) 논문에서 정의한 안전자산에 의하면 시장이 극단적으로 하락하는 시기에 위험자산(주식)과 음의 상관관계 또는 무상관인 자산
위험자산과 방향이 다르게 가는가를 검증


## 이벤트 스터디, GARCH, 분위수 회귀 관련 내용

### 이벤트 스터디 : CAR 기반 수익률 반응, 위기 때 수익률이 올랐는가를 보는 보조 증거
질문: 위기 이벤트 발생 전후 BTC 수익률이 양수였는가?
      → 위기 때 수익률이 오른다면 Safe Haven 신호
한계: 주식과의 관계를 직접 보지 않음

### GARCH : 지정학 리스크와 변동성 관계 확인용
질문: 지정학 리스크가 높을수록 BTC 변동성이 커지는가?
      → 안전자산 여부보다는 "BTC가 위험자산처럼 불안정해지는가"를 보는 것
      → 보조 분석
<공식>
σ²(t) = ω + α·ε²(t-1) + β·σ²(t-1)
σ²(t) : t일의 조건부 분산 (변동성²)
ω : 기저 변동성 (상수)
α·ε²(t-1): 전일 충격의 영향 (ARCH 효과)
β·σ²(t-1): 전일 변동성의 지속성 (GARCH 효과)

<적용 모델>
σ²(t) = ω + α·ε²(t-1) + β·σ²(t-1) + γ·X(t)   ← GARCH-X (우리 모델)
γ·X(t) : 외생변수의 추가 설명력
X(t) : GPR_custom, VIX, Fear&Greed
γ > 0 & 유의 → 지정학 리스크가 BTC 변동성을 키움

### 분위수 회귀 (주요 분석) : 극단 하락 시 BTC 방향성
질문: 주식이 폭락하는 하위 5~10% 구간에서
      BTC가 반대 방향으로 움직였는가? (β < 0)
      → 이게 Baur & Lucey 안전자산 정의에 가장 직접적으로 대응

β < 0 (τ=0.05, 0.10에서) & 유의
→ 주식이 폭락할 때 BTC는 반대로 움직임
→ Safe Haven

β > 0 (τ=0.05, 0.10에서) & 유의
→ 주식이 폭락할 때 BTC도 같이 하락
→ Risky Asset

### 분석 결과
Safe Haven : 주식 폭락 시 수익률이 유지되거나 오히려 오름 (β < 0)
Hedge : 평상시에도 주식과 반대 방향 (장기 음의 상관관계)
Diversifier : 주식과 상관관계가 낮지만, 폭락 시에도 같이 떨어질 수 있음



## 수정 필요사항
- returns.csv 시작일 문제 → hormuz_crisis 4거래일
현재 returns.csv가 2019-07-02부터 시작
hormuz_crisis 수집 기간은 2019-05-18~ 이므로 대부분 미커버
분석에 포함된 거래일이 4일뿐 → 이벤트 스터디·분위수 회귀 모두 제외됨

- 실제 arch 라이브러리로 돌린 내용
arch 라이브러리가 없는 환경에서 OLS 근사로만 돌림
OLS 근사 결과는 실제 GARCH와 다를 수 있어 신뢰도 낮음
γ(GPR_custom) 유의성이 미확인 상태 → 3종 분석 중 1개 공백

- 이벤트 스터디 코드 — 이벤트 창 ±25 → ±17 수정 필요
기존 코드(이벤트_스터디.ipynb v1)는 EVENT_WINDOW = 25로 설정
master_data가 기준일 전후 최대 20일 → 창이 잘려 데이터 손실
v2(이벤트_스터디_v2.ipynb)에서 ±17로 수정했지만, 실제 실행 파일 확인 필요

- gpr_combined.csv의 F3 공식이 절대 보도량 / 상대 보도량 확인
초기 F3 공식: −mean_tone × log(1+N)  ← 절대 보도량
수정된 F3 공식: −mean_tone × log(1+N/N̄)  ← 상대 보도량
현재 gpr_combined.csv가 어느 버전으로 만들어졌는지 확인 필요

- 분위수 회귀 유의성 검정 미실시
현재 β 계수의 부호(±)만 확인 → p값 없이 Safe Haven 판단
statsmodels QuantReg를 사용해야 신뢰구간 + p값 확인 가능
부호만으로는 통계적으로 유의하지 않을 수 있음

- Bootstrap p값 계산 로직 재확인
추정 기간 AR과 이벤트 창 AR을 올바르게 구분해서 bootstrap 돌렸는지 확인
t-test p값과 bootstrap p값이 크게 다르면 로직 오류 가능성
